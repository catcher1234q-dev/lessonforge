# Founder Review Checklist

## Before Founder Review
- Run `npm run verify:focused` before asking for feedback on buyer, seller, admin, or founder-facing flows.
- Run `npm run verify:app` before a broader review pass when shared logic and browser flows both changed.
- Use the fuller verification guide in [docs/verification-checklist.md](/Users/mikhailtripp/Documents/New%20project/docs/verification-checklist.md).
- Use [docs/live-readiness-status.md](/Users/mikhailtripp/Documents/New%20project/docs/live-readiness-status.md) when you need the plain-English answer to “what works now” versus “what still needs keys.”

## What To Review Right Now
- Does the marketplace feel trustworthy before purchase?
- Does the seller dashboard make it obvious what to fix next?
- Do shortlist, storefront, and product pages feel connected instead of isolated?
- Do launch-ready listings feel clearly different from incomplete ones?
- Does the admin/founder view explain marketplace health in plain language?

## What To Review At The End Of Each Phase
- Does the app still feel simple and premium?
- Is the seller flow easy to understand?
- Is the buyer flow trustworthy?
- Are the plan and credit limits still sensible for margins?
- Are we accidentally building too much too early?

## What You Should Usually Comment On
- wording
- layout clarity
- trust signals
- whether a flow feels confusing
- pricing and packaging concerns
- anything that feels too technical or too busy

## What You Usually Do Not Need To Decide
- internal code structure
- low-level database details
- deployment mechanics
- how API routes are organized

## Current Founder Checkpoints
- Homepage and marketplace trust signals
- Product page buy-box clarity
- Seller storefront credibility
- Seller dashboard “needs action” and “top performer” views
- Admin and founder health summaries

## Moments That Do Need Founder Input
- pricing changes
- plan packaging
- homepage messaging
- seller and buyer trust positioning
- refund policy choices
- launch scope tradeoffs
